3. What stays opinionated vs becomes template

The cut between "ship as-is, adopt without editing" and "ship as scaffolding, adopter fills in" is the architectural heart of the deposit. Items inside conventions/ are split as follows.

Opinionated (ship as-is)

FileWhy it stays opinionated
research.mdSource-faithfulness contract, citation accuracy rules, lit-search QC, anti-hallucination. The hardest-won safeguard in the framework. KB ingest workflows mandatorily load this
iteration-workflow.mdThe six-phase research-iterate loop. Structural pattern is general across domains
research-quality-gates.mdAnalytic / visual / literature / framing gates. Generic acceptance criteria
visual-review-protocol.mdRender-and-read protocol for figures. Cross-disciplinary utility
readiness-assessment.mdPhase 0 expertise-coverage check
system-improvement-protocol.mdHow the framework updates itself. Self-referential, reusable as-is

Templates (adopter fills)

FileWhat template means here
voice.template.mdStructured slots: register, hedging calibration, banned words, punctuation rules, sentence rhythm, signature phrases. Each slot has 1-2 sentences of guidance plus a worked example from a fictional adopter lab
manuscript-format.template.mdIMRAD skeleton; section-by-section slots for "what your lab puts here" with prompts
reply-format.template.mdReviewer-response skeleton: comment-typing taxonomy slot, tone slot, push-back conventions slot
figure-format.template.mdTooling-agnostic skeleton; palette, typography, export rules
code-format.template.mdProject structure, naming, performance, testing slots. Avoid R-vs-Python prescription
goal-spec.template.mdPer-project endpoint definition (was in core/)
agents/_domain-specialist.template.mdRole skeleton + one worked example (a fictional non-fisheries specialist)
knowledge_base/_topic.template/One placeholder topic showing INDEX.md structure and one example article. No real content

Naming convention: *.template.md makes the distinction visible in ls output. Adopter populates voice.template.md and saves as voice.md. Skill loaders look for the populated version first; fall back to a "configure voice via setup" message that points to setup/SKILL.md.

4. The adopter experience

Component-level READMEs

Each of the six top-level folders gets a README of roughly 1-2 pages with a fixed five-section structure:

  1. What this folder is for. One paragraph tying it to the §3 architectural fundamental from the paper (or to its supporting role for those fundamentals)
  2. How it is structured. File-by-file map with a one-line purpose for each
  3. How to populate. Pointer to setup/SKILL.md for AI-assisted, plus manual instructions for the advanced path
  4. What good looks like. Worked example (~10 lines of a populated file) so the adopter has a concrete target
  5. How it connects. List of cross-references at runtime (e.g., which skills load this convention)

Setup interview flow

Two routes to populate the templates:

  1. HTML form (setup/lab-onboarding.html): single self-contained HTML, localStorage-persisted, export-to-text button. Sections: lab identity, voice (paste sample paragraphs, set preferences), manuscript norms, code stack, agent roster, KB seed topics, quality preferences. Adopter fills the form, exports the result, pastes into any LLM with the printed prompt block
  2. Chat-driven (setup/SKILL.md): the same questions delivered as an interview by the LLM, for adopters who prefer chat over a browser form

Both routes terminate in the same outcome: the LLM uses the answers as input to setup/SKILL.md, which generates populated versions of voice.md, manuscript-format.md, code-format.md, etc., plus stubs for agents/<domain-specialist>.md files based on the interview answers. The adopter reviews, edits, commits.

Target: fork-to-running in roughly one focused hour, compared with a day-or-more manual carve.
